# 既知の制約と懸念点

> **このドキュメントの目的**: 個人プロジェクトの限界を誠実に開示し、何が妥協で何が将来の改善対象なのかを明示する。
> ポートフォリオとしては「完璧でない部分を理解しているか」が問われると考え、敢えて詳細に列挙しています。

各項目は **影響範囲 / 現状の妥協 / 改善案** の三点セットで整理しています。

---

## 1. データ完全性 (Data Integrity)

### 1.1 解答時間がクライアント計測

- **影響範囲**: `fact_answer.time_sec`
- **現状の妥協**: ブラウザの `performance.now()` で計測しているため、ユーザーが意図的に・あるいは別タブを開いて中断した場合に値が乱れる。
- **改善案**:
  - サーバ側で「問題表示時刻」を発行し、解答送信時の `now() - issued_at` を真値として記録
  - ただし個人利用では自分が改ざんしないことを前提にしているため、現状は許容

### 1.2 正誤の自己申告バイアス

- **影響範囲**: `is_correct`
- **現状の妥協**: ユーザーが解答を見た後にチェックを入れるため、「正解を見てから自信あり」と入力するリスク
- **改善案**:
  - 問題マスタを構造化し、選択肢方式にして即時判定
  - ただしマスタ整備のコストが学習時間を圧迫するためPhase 2扱い

### 1.3 自信度のキャリブレーション

- **影響範囲**: `confidence`
- **現状の妥協**: Likert 1〜5の主観値。自信過剰／過少が個人差として混入
- **改善案**:
  - キャリブレーションプロット（confidence別の正解率）を分析側で監視
  - 一定期間後に確信度のキャリブレーション関数を学習・補正

## 2. 統計的妥当性 (Statistical Validity)

### 2.1 解答時間の分布仮定

- **影響範囲**: `analysis/anova_weak_topics.py`
- **現状の妥協**: 一元配置ANOVAを用いているが、**解答時間は右に歪んだ対数正規分布**になることが多く、ANOVAの正規性仮定を満たさない可能性が高い
- **改善案**:
  - **Kruskal-Wallis検定**へ切替（ノンパラメトリック）
  - もしくは `log(time_sec)` で変換してからANOVA
  - 現状は両方の結果を出すユーティリティを追加するのがコスト対効果◎

### 2.2 線形回帰によるスコア予測

- **影響範囲**: `analysis/score_projection.py`
- **現状の妥協**: 学習効果は本来 **学習曲線（対数）** で進むことが多い。線形仮定では序盤を過小・終盤を過大に予測する
- **改善案**:
  - `log(weeks)` を説明変数に
  - データが増えたら3次スプライン or ガウス過程回帰
  - **試験本番までデータ点数が10〜15週分しかない**ため複雑モデルは過学習リスク。**素朴な線形でも十分に意思決定材料になる**判断で現状は維持

### 2.3 出題範囲の重み (`dim_topic.weight`) の妥当性

- **影響範囲**: 期待スコア計算
- **現状の妥協**: 出題範囲表PDFの ●/〇 を 1.0 / 0.5 でハードコードしただけ。実際の配点重みとは異なる可能性
- **改善案**:
  - 過去問の小問単位で配点を集計してデータ駆動化
  - `data/seed/topics.yaml` を分離し、weight更新を独立コミットに

### 2.4 IQRによる外れ値除外

- **影響範囲**: `time_outlier_flag`
- **現状の妥協**: ユーザー × 分野ごとにIQRを毎回計算。データが少ない初期は四分位そのものが不安定で、**正常値が外れ値判定される**ケースがある
- **改善案**:
  - 件数が一定以上溜まるまでフラグを立てない
  - 個人内ではなくグローバルベースラインを併用

## 3. セキュリティ (Security)

### 3.1 `--allow-unauthenticated` での公開

- **影響範囲**: Cloud Functions エンドポイント
- **現状の妥協**: id_token検証だけが認証層。レート制限なし
- **改善案**:
  - **Cloud Armor** でレート制限・地理的制限
  - 1ユーザー1分100リクエストのスロットル
  - 異常検知ログを Cloud Logging で集計

### 3.2 LINE Verify API依存

- **影響範囲**: 全エンドポイントの可用性
- **現状の妥協**: 毎リクエストで `api.line.me/oauth2/v2.1/verify` をコール。LINE側障害時にアプリ全停止
- **改善案**:
  - **JWT署名検証** に切替（LINEのJWKSを使用）
  - 検証結果を短期キャッシュ（例: 5分のメモリキャッシュ）

### 3.3 `id_token` の有効期限管理

- **影響範囲**: フロントエンドのセッション
- **現状の妥協**: クライアントは初回 `getIDToken()` の値を保持し続ける。トークンは ~1時間で期限切れ
- **改善案**:
  - `liff.getIDToken()` を都度呼び直す（LIFFが内部で再発行）
  - 401受信時の自動リトライロジック追加

### 3.4 Salt のローテーション戦略不在

- **影響範囲**: `user_id_hash`
- **現状の妥協**: `USER_ID_SALT` を環境変数で固定。ローテーションすると過去ログとjoinできなくなる
- **改善案**:
  - Salt のバージョンを保持し、再ハッシュバッチを組む
  - Secret Manager に格納し、IAM経由で取得

### 3.5 CORS設定の単純さ

- **影響範囲**: ブラウザからのアクセス制御
- **現状の妥協**: `ALLOWED_ORIGIN` を1つだけ環境変数指定
- **改善案**:
  - ステージング環境ができた時点で複数オリジン対応のリスト化
  - Origin検証 + プリフライトキャッシュ調整

## 4. スケーラビリティ・コスト

### 4.1 単一ユーザー前提の設計

- **影響範囲**: `dim_user`、ETL
- **現状の妥協**: 自分専用想定なのでユーザー登録フロー・テナント分離なし
- **改善案**:
  - `dim_user` の自動upsert (Cloud Functions側で初回ログイン時に登録)
  - BigQueryの **Row-Level Security** をユーザーごとに

### 4.2 ETL Scheduled Queryのフルスキャン傾向

- **影響範囲**: BigQuery コスト
- **現状の妥協**: IQR計算で全データスキャン（パーティションは効くが、ユーザー × 分野ごとに集計するため広い）
- **改善案**:
  - IQRをマテリアライズドビューに切り替え
  - dbtで incremental モデル化し、過去30日のみ再計算

### 4.3 Cloud Functions Gen2 のコールドスタート

- **影響範囲**: 初回送信時のレイテンシ
- **現状の妥協**: 256Mi / Gen2 でコールドスタート2〜3秒
- **改善案**:
  - `min-instances=1` で常時稼働化（個人プランでも月数百円程度）
  - Cloud Run と同等の単価のため、コスト試算で判断

## 5. 観測性 (Observability)

### 5.1 構造化ログの不在

- **影響範囲**: 障害分析
- **現状の妥協**: `logging.info` でテキストログのみ
- **改善案**:
  - `google-cloud-logging` の `setup_logging()` で JSON Payload に
  - `trace_id` をリクエストごとに付与し分散トレーシング

### 5.2 SLI/SLOの未定義

- **影響範囲**: 運用判断
- **現状の妥協**: 個人利用なので明示的SLO不要と判断
- **改善案**: 仮にSaaS化するなら以下を設定
  - SLI: 送信API成功率99%、p95レイテンシ1秒
  - SLO: 月次99.5%

### 5.3 データ品質モニタリング

- **影響範囲**: 分析の信頼性
- **現状の妥協**: スキーマ違反やNULL率の異常を検出する仕組みなし
- **改善案**:
  - **Great Expectations** または **Soda** をETL後に走らせる
  - 主要カラムの欠損率/分布シフトをSlack通知

---

## 6. 保守性 (Maintainability)

### 6.1 dbt未使用

- **影響範囲**: SQLの再利用性、依存関係の可視化
- **現状の妥協**: SQLファイルを順序付きで実行する素朴な構成
- **改善案**:
  - **dbt-bigquery** を導入し、`models/`, `tests/`, `docs/` 構成に
  - 系譜（lineage）グラフを自動生成

### 6.2 Terraform未使用

- **影響範囲**: 環境再現性
- **現状の妥協**: `gcloud` コマンドのシェルスクリプト
- **改善案**:
  - `infra/terraform/` 配下に空のmoduleとREADMEを置いてある（Phase 2で実装）
  - `terraform-google-modules` を流用すれば1日で移行可能

### 6.3 マイグレーション戦略の不在

- **影響範囲**: スキーマ進化
- **現状の妥協**: テーブル追加は `CREATE TABLE IF NOT EXISTS`、カラム追加は手動 `ALTER`
- **改善案**:
  - dbt導入後は `on_schema_change` 設定で自動対応
  - もしくは **Liquibase** や **Flyway** をBigQuery対応で

## 7. テスト・CI/CD

### 7.1 テストカバレッジが最小限

- **影響範囲**: バリデーションロジック
- **現状の妥協**: `backend/tests/` にバリデーションのpytestのみ
- **改善案**:
  - 統合テスト（BigQuery emulatorまたはdry-run）
  - フロントエンドの Playwright E2E
  - SQLの dbt test 化（unique, not_null, accepted_values）

### 7.2 SQLの構文チェックなし

- **影響範囲**: デプロイ時の事故
- **現状の妥協**: 実行するまで構文エラーがわからない
- **改善案**:
  - `bq query --dry_run` をCIに組込
  - **sqlfluff** で linting

## 8. アクセシビリティ・UX

### 8.1 LIFFの動作環境制約

- **影響範囲**: ユーザー体験
- **現状の妥協**: LINE内ブラウザを前提。外部ブラウザでも動くが LINE Login 経由が必要
- **改善案**:
  - PWA化してホーム画面追加対応
  - オフラインキャッシュ強化（Service Worker）

### 8.2 UIのアクセシビリティ未検証

- **影響範囲**: スクリーンリーダー対応
- **現状の妥協**: `aria-*` 属性最小限、コントラスト比未測定
- **改善案**:
  - axe-core でCI上の自動チェック
  - WCAG 2.1 AA準拠を目標

## 9. プロダクト・運用

### 9.1 バックアップとリカバリ計画

- **影響範囲**: データ消失リスク
- **現状の妥協**: BigQuery の time-travel (7日) のみ。raw削除→mart再構築のリハーサル未実施
- **改善案**:
  - `bq extract` での日次GCSバックアップ
  - DR手順書の整備

### 9.2 出題範囲改訂への追従

- **影響範囲**: 2025年から CBT 範囲が一部更新されている
- **現状の妥協**: 2024年時点の範囲表をシード化
- **改善案**:
  - `data/seed/topics.yaml` を年度ごとにバージョン管理
  - dim_topic に `effective_from` カラムを追加

---

## まとめ: 個人利用とポートフォリオの両立

このプロジェクトは「**自分の合格に必要な最小機能**」と「**他者に見せられる設計品質**」のバランスで設計されています。
すべての懸念に対応せず、**何を妥協し、何を将来に回したかを明示する**ことが、データエンジニアとしての判断力の証明になると考えています。

「完璧なシステム」より「**意図的に不完全で、不完全さを把握しているシステム**」のほうが運用上は信頼できる、というのが本プロジェクトの裏テーマです。
